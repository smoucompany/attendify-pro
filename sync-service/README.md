# Attendify Pro — Device Sync Service

خدمة مستقلة تعمل بشكل مستمر **محلياً داخل شبكة الفرع** (نفس شبكة جهاز البصمة)،
تتصل بجهاز/أجهزة ZKTeco عبر TCP:4370، وتزامن سجلات الحضور مع Attendify Pro backend.

## لماذا خدمة منفصلة؟

الموقع منشور على Vercel كـ serverless functions، ولا يمكن لهذه الدوال الاحتفاظ
باتصال TCP دائم مع جهاز داخل شبكة محلية. لذلك تعمل هذه الخدمة على أي جهاز
كمبيوتر/سيرفر صغير (Mini-PC) موجود فعلياً داخل الفرع، وتتواصل مع الـ backend
عبر HTTPS العادي.

## التشغيل

```bash
cd sync-service
npm install
cp .env.example .env          # عدّل BACKEND_URL و SYNC_API_KEY
cp devices.json.example devices.json   # عدّل IP/Port لكل جهاز (id يطابق صف الجهاز في لوحة التحكم)
npm start
```

للتشغيل الدائم (يعيد التشغيل تلقائياً عند الأعطال أو إعادة تشغيل الويندوز)، استخدم PM2:

```bash
npm install -g pm2
pm2 start src/index.js --name attendify-sync
pm2 save
pm2 startup   # يضبط تشغيل تلقائي مع بدء تشغيل الويندوز
```

## متغيرات البيئة (`.env`)

| المتغير | الوصف |
|---|---|
| `BACKEND_URL` | رابط موقع Attendify Pro (Vercel) |
| `SYNC_API_KEY` | يجب أن يطابق `DEVICE_SYNC_API_KEY` في متغيرات بيئة Vercel |
| `POLL_INTERVAL_MS` | فاصل المزامنة (افتراضي 30000 = 30 ثانية) |
| `DEVICES_CONFIG` | مسار ملف تعريف الأجهزة (افتراضي `./devices.json`) |

## `devices.json`

كل جهاز فعلي في `devices.json` يجب أن يطابق `id` صف مقابل في جدول `devices`
(الذي تُضيفه من لوحة "أجهزة البصمة" داخل التطبيق):

```json
[
  { "id": "dev-1730000000000-1", "ip": "192.168.1.201", "port": 4370 }
]
```

يدعم الملف عدة أجهزة في نفس الوقت — كل جهاز له دورة مزامنة مستقلة، وفشل جهاز
واحد لا يوقف مزامنة البقية.

## ماذا تفعل كل دورة مزامنة؟

1. الاتصال بالجهاز (مع إعادة محاولة تلقائية بتأخير تصاعدي: 5ث → 10ث → 30ث)
2. تنفيذ أي أوامر معلّقة من اللوحة (Sync Now / Test Connection / Restart)
3. جلب سجلات البصمة الجديدة فقط (منذ آخر مزامنة، محفوظة في `state.json` محلياً)
4. رفعها للـ backend (`/api/devices/:id/attendance-bulk`) — يُدمج كل يوم كأول/آخر بصمة (دخول/خروج)
5. رفع heartbeat + سجل مزامنة، ثم قطع الاتصال حتى الدورة القادمة

## ربط الموظف بمعرّفه على الجهاز

مطابقة سجلات البصمة بالموظفين تتم عبر حقل `deviceUserId` في بيانات الموظف
(`employees.data.deviceUserId`) — وهو رقم المستخدم الداخلي على جهاز ZKTeco،
وليس بالضرورة نفس الرقم الوظيفي. أضِف هذا الحقل يدوياً حالياً عند إعداد الموظف
لأول مرة (سيُضاف حقل مخصص له في واجهة الموظفين في مرحلة لاحقة).

## السجلات (Logs)

تُكتب سجلات التشغيل في `logs/YYYY-MM-DD.log` بالإضافة إلى الطرفية (console).
أخطاء الاتصال/المزامنة تُرفع أيضاً إلى جدول `sync_errors` في لوحة التحكم
(صفحة "أجهزة البصمة" ← السجلات).
